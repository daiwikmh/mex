# Project: Nocturne / "Lend your bank history. Never show it."

A private credit data co-op for DeFi lending. The product framing is locked in `../plan.md` and `../build-plan.md` — read those before pivoting product scope. The four load-bearing pillars are **AI** (LLM-in-TEE underwriting), **Data** (yield-bearing private vaults), **DeFi** (undercollateralized USDC against a ZK score), and **Midnight** (shielded cohort + score proof + private royalties). Each is required; remove any one and the moat collapses.

## Current state (2026-05-16)

**What's real:**
- Next.js 16 App Router + React 19 + Tailwind 4 (using semantic theme tokens, not raw colors)
- Five-section stacked landing (`Hero`, `Features`, `Stats`, `Testimonials`, `CTAFooter`) via `StackedSection.tsx`
- Two app routes — `/app/vault` (borrower/contributor portal) and `/app/lend` (mock lender portal)
- **Typed financial knowledge graph** extracted from uploaded bank CSVs (deterministic parser, no LLM yet — see `lib/graph.ts`)
- Graph dashboard at `/app/vault` → "Knowledge graph" tab — SVG-rendered column layout with click-to-inspect node + edge detail panels
- `localStorage` persistence for vault state, underwriting result, and the graph

**What's still mocked (`lib/mock.ts`):**
- "Encryption" — just hashes a label, no real AES-GCM yet
- TEE underwriting — `mockUnderwriting()` derives a score deterministically from a hash, not from the graph (Phase D will fix this)
- Midnight registration — commitment is `shortHash("commit:...")`, no real chain call
- ZK proof — `reasoningHash` is a string, no proof artifact
- Cohort — `cohortPubkeys` is generated by hashing; no real cohort selection

## Visual identity

Theme pivoted on 2026-05-17 from Fundable yellow → **Spade-inspired pale sage + forest green + near-black**. The yellow read as punchy-startup; the sage palette reads as serious-fintech, which is the right signal for a private credit product. Reference image in `public/image.png` (Spade landing). Defined in `app/globals.css`:

```css
--background: #e8f0d8   /* pale sage  */
--foreground: #0d0d0d   /* near-black */
--accent:     #0f5a37   /* forest green (CTAs, accents only) */
--surface:    #f4f6e6   /* warm off-white cards */
--surface-2:  #eef2dc
```

**Landing section contrast rhythm** (max contrast between adjacent panels):

```
Hero          #e8f0d8   pale sage    ← brand entry
Features      #0f5a37   forest green ← inverted, dramatic
Stats         #f7f9ef   off-white    ← calm, lets numbers breathe
Testimonials  #0d0d0d   near-black   ← weighty quotes
CTAFooter     #e8f0d8   pale sage    ← bookend
```

The contrast pattern (sage → forest → cream → black → sage) is the key reason this landing reads as a real fintech product. Don't unify the section colors.

Hero uses **serif headline** (Georgia/system serif via `.font-serif` utility) with a scattered icon cluster on the right — see `components/landing/Hero.tsx`'s `IconCluster` for the chip-positioning convention (9 chips, varying sizes 50–76px). Chip palette is now restrained sage/forest/black/cream + one terracotta accent (`#c97a4a`) for warmth. No saturated blues or bright greens — those were Fundable-era choices and now look wrong.

**Important on the section colors:** the components use **inline `style={}` overrides**, NOT Tailwind theme tokens, because Tailwind tokens swap globally. If you re-theme, the explicit hex codes in each section file are the source of truth, not `globals.css`.

## Typed financial graph schema (`lib/graph.ts`)

**Entities (6):**
- `account` — bank account (Checking, Savings)
- `transaction` — individual line item
- `merchant` — deduplicated by normalized description key
- `income_stream` — recurring positive payments (≥3 hits, all positive)
- `debt_obligation` — recurring negative to creditor names (mortgage / loan / credit / amex etc.)
- `recurring_charge` — other recurring negatives (subscriptions, utilities)

**Edges (6):**
- `paid_to` — Transaction → Merchant (amount < 0)
- `earned_from` — Transaction → Merchant (amount > 0)
- `belongs_to` — Transaction → Account
- `represents` — Transaction → IncomeStream | DebtObligation | RecurringCharge
- `recurring_with` — Merchant → IncomeStream | DebtObligation | RecurringCharge
- `overdrew_at` — Account → Transaction (reserved, not emitted yet)

The CSV parser auto-detects columns via header aliases (`date | posted | posting date`, `description | name | merchant | memo`, `amount | amt | value`, etc.). Plaid exports work without configuration.

## Frame0 patterns ported here

This project borrows the architecture of [Frame0](https://github.com/daiwikmh/Frame0) — a private intelligence platform built end-to-end on 0G. The transferred patterns are:

1. **Typed knowledge graph from heterogeneous sources** — Frame0 extracts `token / protocol / company / person / jurisdiction / event / topic` entities from news; Nocturne extracts `account / transaction / merchant / income_stream / debt_obligation / recurring_charge` from bank statements. Same architectural shape, different domain.
2. **Domain/range-validated typed edges** — same pattern; only certain edge types are valid between certain entity types.
3. **Click-to-inspect dashboard with context panel** — node click → attributes + connected relationships in verb-phrase form ("this earned from Stripe payroll").
4. **Source artifacts persisted as graph nodes** — Frame0 makes articles graph nodes; Nocturne makes transactions graph nodes, so even sparse extractions stay navigable.
5. **Audit log with provenance** — every relationship carries `evidence` (the raw CSV row) + `observed_at` (date), mirroring Frame0's per-edge evidence quote.

## Hackathon constraints to respect

- **Ship the visible loop first; gate later.** Every protection layer added in advance becomes a debugging surface during the demo. Keep encryption / chain / TEE mocked while the *user-facing flow* is being polished.
- **Hero, graph view, ResultPanel** are the three surfaces a judge sees first. Disproportionate polish budget belongs there.
- **The graph IS the privacy proof, visually.** When a judge asks "but how do I know the AI used my real data?" the answer is the graph view — the entities and relationships are visibly derived from the upload. Without the graph, the privacy claim feels abstract.
- The original product framing — Private KYC + selective disclosure — was considered and rejected (see `../plan.md`). Don't re-pivot to it without strong reason; the credit co-op has clearer buyers and a quantitative output that's much richer to ZK-prove than a binary jurisdiction check.
